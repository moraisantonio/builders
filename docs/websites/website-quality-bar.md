# Website Quality Bar — O que define um site pronto para a Builders

> Este documento define o padrão mínimo que todo site entregue pela Builders precisa atingir.
> Use antes de apresentar qualquer entrega ao cliente.

---

## A pergunta central

**Em 5 segundos, um visitante que nunca ouviu falar do negócio consegue responder:**
1. O que este negócio faz?
2. Para quem?
3. O que eu faço agora?

Se a resposta for não para qualquer uma dessas perguntas, o site não está pronto.

---

## Barra de estratégia

- [ ] O objetivo principal do site está claro e é mensurável
- [ ] Existe um único CTA principal por página — não três opções concorrentes
- [ ] A proposta de valor aparece above the fold, sem precisar rolar
- [ ] Cada seção tem uma função: converter, informar ou construir confiança
- [ ] Nenhuma seção existe só por convenção — se não tem função, foi cortada

---

## Barra de copy

- [ ] Nenhum texto usa placeholder ("Lorem ipsum", "Texto aqui", "Em breve")
- [ ] A headline principal tem no máximo 8 palavras
- [ ] O CTA principal usa verbo de ação + resultado (não "Saiba mais")
- [ ] Nenhuma frase tem mais de duas linhas no mobile
- [ ] Não há jargão técnico que o cliente do cliente não entenderia
- [ ] Não há clichês: "solução completa", "ecossistema", "potencializar", "alavancar"
- [ ] O texto soa como o negócio específico, não como qualquer empresa

---

## Barra visual

- [ ] Hierarquia clara: headline → benefício → prova → ação
- [ ] Contraste mínimo 4,5:1 entre texto e fundo (verificado com ferramenta)
- [ ] Não foram usados gradientes decorativos
- [ ] Não há sombras pesadas em cards
- [ ] Não há três colunas genéricas com ícones repetidos
- [ ] Não há mais de três fontes diferentes na mesma página
- [ ] Imagens são fotos reais ou contextualizadas — não stock photo genérico
- [ ] Espaçamento consistente entre seções (múltiplos de 8px)

---

## Barra técnica

- [ ] O site abre em menos de 3 segundos em conexão 4G (Lighthouse > 70 performance)
- [ ] Todas as imagens estão em WebP ou AVIF
- [ ] O site funciona em iPhone (iOS Safari) e Android (Chrome)
- [ ] Todos os links funcionam — nenhum 404
- [ ] Todos os formulários enviam e o cliente recebe a notificação
- [ ] O botão/link de WhatsApp abre o número correto

---

## Barra de SEO

- [ ] Cada página tem title tag único (50–60 caracteres)
- [ ] Cada página tem meta description única (140–160 caracteres)
- [ ] Existe um único h1 por página com a keyword principal
- [ ] Google Search Console configurado e verificado
- [ ] Sitemap.xml criado e submetido
- [ ] Schema LocalBusiness configurado (para PMEs locais)

---

## Barra de analytics

- [ ] Google Analytics 4 instalado e enviando dados
- [ ] Evento de clique no CTA principal rastreado
- [ ] Clique no WhatsApp rastreado
- [ ] Envio de formulário rastreado

---

## Barra de acessibilidade

- [ ] Alt text em todas as imagens informativas
- [ ] Hierarquia de headings correta (h1 único, h2 por seção, h3 para subseções)
- [ ] Botões com label descritivo
- [ ] Site navegável por teclado

---

## Barra de documentação

- [ ] `10-launch-checklist.md` do projeto preenchido e aprovado
- [ ] Stack e decisões principais registradas no arquivo do cliente
- [ ] Credenciais de acesso entregues ao cliente de forma segura
- [ ] Próximos passos definidos e comunicados

---

## O que fazer se um item falhar

Não entregar. Corrigir primeiro.

Se for um item negociado intencionalmente (ex: analytics adiado por pedido do cliente), registrar no `11-post-launch-notes.md` como pendência conhecida com data para resolver.
